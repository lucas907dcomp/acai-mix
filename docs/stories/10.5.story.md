## User Story

**Como** administrador do AçaiMix,
**Quero** ver no Dashboard a receita separada por produto e no histórico de vendas uma coluna de produto com filtro,
**Para que** eu saiba exatamente quanto cada produto rendeu no período e possa auditar vendas de qualquer produto (incluindo casquinhas e produtos avulsos).

## Acceptance Criteria

- [x] AC1: **Dashboard — novo card/seção "Vendas por produto"** mostrando, para o período filtrado (hoje/semana/mês):
  - Tabela com colunas: Produto, Qtd vendida, Receita (R$), % do total.
  - Linhas ordenadas por Receita DESC.
  - Linha extra "Casquinha (adicional)" mostrando count de vendas com `has_casquinha=true` e receita extra (count * R$ 1,00).
  - Apenas vendas com `status = 'COMPLETED'` + NULL (retrocompat).
- [x] AC2: Cards existentes do dashboard (receita do dia/semana/mês) **continuam corretos**: somam `sales.amount` de vendas `COMPLETED`. Nenhuma regressão nos valores exibidos — validar que os valores não mudaram para vendas históricas de açaí.

- [ ] AC9: Query agregada do dashboard "Vendas por produto" para 30 dias com até 30.000 vendas responde em < 500ms. Usar índice `idx_sales_product_id` (da Story 10.2) e join com `products`. Implementar via view ou função RPC Supabase se necessário.

## Dev Notes

**Não tocar nas queries dos cards existentes do Dashboard (AC2):** Os cards de "Receita do dia", "Receita da semana", etc. usam suas próprias queries. Esta story adiciona uma nova seção — não modifica as existentes. Separar claramente o código novo do existente para minimizar risco de regressão.

**JOIN com products no histórico:** O Supabase JS client suporta foreign table joins:
```typescript
supabase.from('sales').select('*, products(name, product_type)')
```
Isso funciona porque `sales.product_id` é FK para `products.id`. O resultado terá `sale.products.name` (pode ser null para vendas antigas sem product_id — tratar com `?? 'Açaí'` ou buscar o nome via fallback).

**Vendas antigas sem `product_id`:** Após o backfill da Story 10.2, todas as vendas devem ter `product_id`. Mas em ambiente de desenvolvimento pré-migration, pode haver nulls. Sempre usar `sale.products?.name ?? 'Açaí'` defensivamente.

**RPC vs View para "Vendas por produto":** A RPC é preferível porque:
1. O período é dinâmico (hoje / 7 dias / 30 dias) — a RPC recebe `p_from` e `p_to` como parâmetros
2. RLS é aplicado via parâmetro `p_location_id` (mais seguro que filtrar na view)
3. Permite EXPLAIN ANALYZE para validar uso do índice `idx_sales_product_id`

**Casquinha como linha separada (AC1):** A casquinha não é um produto no catálogo (`product_type = 'unit'`) — é um adicional de vendas de açaí. Para ela aparecer no dashboard como "Casquinha (adicional)", fazer query separada:
```sql
SELECT 'Casquinha (adicional)' AS product_name, 
       COUNT(*) AS qty, 
       COUNT(*) * 1.00 AS total_amount
FROM sales 
WHERE has_casquinha = true 
  AND location_id = p_location_id 
  AND status = 'COMPLETED'
  AND created_at BETWEEN p_from AND p_to
```
Unir via `UNION ALL` na RPC ou fazer duas queries no hook e combinar no cliente.

**CSV — colunas ao final (AC6, risco R8 do EPIC):** A ordem das colunas existentes é exatamente: `Data/Hora;Peso (g);Preço/g (R$);Valor (R$);Pagamento;Status`. As novas colunas vão após `Status`. O cabeçalho novo será: `Data/Hora;Peso (g);Preço/g (R$);Valor (R$);Pagamento;Status;Produto;Tipo;Quantidade;Casquinha`. Testar no Excel antes do PR.

**Coluna "Peso" robusta a null (AC4):** Para vendas unitárias, `weight_grams` pode ser `0` ou `NULL`. O código de exibição deve ser:
```typescript
const displayWeight = (sale: Sale) => {
  if (!sale.weight_grams || sale.weight_grams === 0) return '—'
  return `${sale.weight_grams}g`
}
```

**Performance AC9:** Verificar com `EXPLAIN ANALYZE` que a query usa `idx_sales_product_id`. Se não usar (ex: para vendas de açaí que têm `product_id` mas a query filtra por `has_casquinha`), analisar se índice adicional em `has_casquinha` é necessário. Para 30.000 vendas em 30 dias, o índice em `product_id` deve ser suficiente.
