## User Story

**Como** administrador da loja,  
**Quero** cancelar uma venda do turno ativo com confirmação explícita,  
**Para que** erros operacionais (peso errado, método de pagamento equivocado, venda duplicada) sejam corrigidos sem corromper os totais financeiros do turno e do dashboard.

## Acceptance Criteria

- [ ] AC1: Botão "Cancelar" visível na tabela de vendas do turno ativo (`ShiftSalesTable`) **somente quando** `profile?.role === 'admin'`
- [ ] AC2: Botão "Cancelar" habilitado somente quando `sale.status === 'COMPLETED'` E `sale.shift_id === activeShift.id`; caso contrário desabilitado com tooltip:
  - `"Apenas vendas do turno ativo podem ser canceladas"` (se `sale.shift_id !== activeShift.id`)
  - `"Esta venda já está cancelada"` (se `sale.status === 'CANCELLED'`)
- [ ] AC3: Ao clicar no botão habilitado, abrir Shadcn `AlertDialog` com:
  - Título: `"Cancelar venda?"`
  - Resumo da venda: data/hora, peso (g), valor (R$), método de pagamento
  - Mensagem: `"Esta ação não pode ser desfeita. A venda será marcada como cancelada e removida dos totais do turno e do dashboard."`
  - Botão "Voltar" (default focus, variante `outline`)
  - Botão "Confirmar cancelamento" (variante `destructive`)
- [ ] AC4: Ao confirmar, executar `UPDATE` no Supabase:
  ```sql
  UPDATE sales SET status='CANCELLED', cancelled_at=NOW(), cancelled_by=auth.uid()
    WHERE id=<saleId>
  ```
  — a RLS `sales_update_cancel_admin` valida `is_admin()` e `can_cancel_sale()` server-side
- [ ] AC5: Após UPDATE bem-sucedido:
  - Toast verde: `"Venda cancelada com sucesso"`
  - `shiftStore.reverseTotals(sale.amount, sale.payment_method)` chamado
  - `queryClient.invalidateQueries({ queryKey: ['shift-sales'] })` — tabela atualiza
  - `queryClient.invalidateQueries({ queryKey: ['sales-history'] })` — histórico atualiza
  - Dialog fecha automaticamente
- [ ] AC6: Se UPDATE falhar (erro 42501 — turno fechou, venda já cancelada, ou staff tentou):
  - Toast vermelho com mensagem amigável: `"Não foi possível cancelar. O turno pode ter sido fechado ou a venda já foi cancelada."`
  - `queryClient.invalidateQueries({ queryKey: ['shift-sales'] })` — forçar refresh para mostrar estado atual
  - **Não chamar** `reverseTotals` (não houve mudança no banco)
- [ ] AC7: Botão "Confirmar cancelamento" desabilitado com spinner enquanto `mutation.isPending === true` (evita cliques duplos)
- [ ] AC8: `ShiftSalesTable.tsx` — o `reduce` que calcula o `total` do rodapé deve filtrar vendas canceladas: `.filter(s => s.status === 'COMPLETED').reduce(sum, s.amount, 0)`
- [ ] AC9: `shiftStore.ts` — nova função `reverseTotals(amount: number, paymentMethod: PaymentMethod): void` adicionada (não alterar `updateTotals` existente)

        total_card: isCard

          : activeShift.total_card,
        total_cash: paymentMethod === 'cash'
          ? Math.max(Math.round((activeShift.total_cash - amount) * 100) / 100, 0)
          : activeShift.total_cash,
      },
    })
  }

## Dev Notes

**`reverseTotals` — NOT `updateTotals`:**  
A função `updateTotals` existente NÃO deve ser modificada. Uma nova função `reverseTotals` separada é criada. Isso é decisão arquitetural explícita da DA-3 — ver `docs/architecture/fase2-tech-decisions.md#4.1`.

**Coerência cliente-servidor:**  
O trigger `trg_update_shift_totals_on_update` (da migration 9.3) já decrementa `shifts.*` no banco quando `status` muda de COMPLETED para CANCELLED. O `reverseTotals` no cliente é otimismo para UX instantânea. O próximo `loadActiveShift()` (polling a cada 60s) vai confirmar os valores. **Não chamar `loadActiveShift` imediatamente após cancelamento** — trust the optimistic update + trigger.

**Race condition AC-F3.8 (turno fecha durante confirmação):**  
Coberto server-side pela `can_cancel_sale()` na RLS: se o turno fechou entre o clique e o UPDATE, a policy retorna false e o UPDATE falha com erro 42501. O `onError` do `useMutation` captura e exibe toast adequado. Não há necessidade de validação client-side adicional para este cenário.

**`cancelled_by` no UPDATE:**  
A RLS `WITH CHECK` força `cancelled_by = auth.uid()`. Portanto, enviar `cancelled_by: profile!.id` no UPDATE é correto e necessário — a policy valida isso. Se `profile!.id !== auth.uid()`, o UPDATE seria rejeitado.

**Shadcn AlertDialog:**  
Padrão já usado no projeto (confirmar em outros componentes). Importar de `@/components/ui/alert-dialog`. Não criar modal customizado.

**Defesa em profundidade no trigger DB:**  
O `GREATEST(... - x, 0)` no trigger do banco previne totais negativos em caso de inconsistência. Turnos com 0 vendas COMPLETED após cancelamento são estado válido e devem ser exibidos normalmente no PDV (sem totais negativos).

**Invalidação de queries após cancelamento:**  
```typescript
queryClient.invalidateQueries({ queryKey: ['shift-sales'] })
queryClient.invalidateQueries({ queryKey: ['sales-history'] })
// dashboard-* NÃO precisa de invalidação manual:
// as views shift_summary e daily_summary já excluem CANCELLED
// e serão atualizadas no próximo refetch natural do dashboard
```
