## User Story

**Como** operador do caixa offline,  
**Quero** que o sistema detecte a virada de turno (16h/23h) mesmo sem internet,  
**Para que** as vendas sejam atribuídas ao turno correto mesmo que o fechamento automático tenha ocorrido na nuvem.

---

## Acceptance Criteria

- [ ] AC1: `syncStore` monitora o relógio local via `setInterval` a cada 30s
- [ ] AC2: Ao atingir 16:00 ou 23:00 no relógio local enquanto offline, executa soft-close
- [ ] AC3: Soft-close: marca `activeShift` como `closed_locally: true` no Dexie e cria turno provisório (`ProvisionalShift`)
- [ ] AC4: Vendas após o horário de corte são associadas ao turno provisório (`local_id`)
- [ ] AC5: Ao reconectar: `drain()` primeiro envia o turno provisório ao Supabase
- [ ] AC6: Após turno provisório ser criado no Supabase, vendas locais têm `shift_id` atualizado para o ID real
- [ ] AC7: Processo transparente para o operador — nenhuma interrupção na tela do PDV

---

## Tasks

### T1 — Monitoramento de horário no syncStore (2h)
- [ ] `startClockWatch()` — `setInterval` a cada 30s
- [ ] Verificar se hora atual `>= 16:00 || >= 23:00` (usando `date-fns`)
- [ ] Só disparar se: `isOnline === false` AND `activeShift.status === 'open'` AND ainda não fechou neste horário
- [ ] Guardar `lastSoftCloseAt` para evitar fechamento duplo no mesmo horário

### T2 — Criar turno provisório no Dexie (2h)
- [ ] `createProvisionalShift(locationId)` → `db.provisional_shifts.add({...})`
- [ ] `local_id = crypto.randomUUID()`
- [ ] `shift_number = activeShift.shift_number + 1`
- [ ] Status: `'provisional'`
- [ ] Atualizar `shiftStore.activeShift` para apontar para o turno provisório (com `local_id`)

### T3 — Drain de turno provisório ao reconectar (2h)
- [ ] Em `syncStore.drain()`: verificar primeiro se há turnos provisórios
- [ ] POST turno provisório para Supabase → receber `remote_id`
- [ ] Atualizar `db.provisional_shifts`: `status: 'synced'`, `remote_id`
- [ ] Atualizar todas as `pending_sales` com `shift_id === local_id` → `shift_id = remote_id`
- [ ] Só então processar as vendas daquela fila

### T4 — Integração com saleStore (0.5h)
- [ ] `saleStore.confirmSale()` usa `activeShift.id || activeShift.local_id` como `shift_id`
- [ ] Venda com `shift_id = local_id` vai para Dexie normalmente

## Dev Notes

**Dual-layer defense:** [Source: docs/architecture/acaipro-system-design.md#8.2]
Esta story implementa a Layer 1 (frontend). A Story 6.4 implementa a Layer 2 (backend).

**Detecção de horário:**
```typescript
import { getHours, getMinutes } from 'date-fns'

function isCutoffTime(): boolean {
  const now = new Date()
  const h = getHours(now)
  const m = getMinutes(now)
  return (h === 16 && m === 0) || (h === 23 && m === 0)
}
```

**Idempotência:** `lastSoftCloseAt` gravado no Dexie (não sessionStorage — sobrevive a refresh). Verificar antes de criar turno provisório.

**Ordem de drain:**
1. Verificar provisional_shifts pendentes
2. POST turno provisório → obter remote_id
3. Atualizar shift_id das pending_sales
4. Drain das pending_sales
