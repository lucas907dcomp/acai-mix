# STORY 10.4: PDV Multi-Produto (Seletor e Fluxo de Venda Unitária)

## User Story

**Como** operador de caixa (admin ou staff) do AçaiMix,
**Quero** ver os produtos avulsos disponíveis no PDV e poder registrar a venda de qualquer um deles com quantidade e método de pagamento,
**Para que** eu possa registrar no sistema 100% das vendas da loja — picolé, água, refrigerante, sorvete — sem precisar de caderno paralelo.

---

## Acceptance Criteria

- [ ] AC1: **Layout do PDV (nova hierarquia visual):**
  - Área principal (dominante na tela): fluxo de açaí por peso INTOCADO — balança, peso, total, toggle casquinha (Story 10.1), botão confirmar. Nenhum step extra adicionado.
  - Área secundária (sidebar ou grid lateral/inferior): cards/botões dos produtos avulsos ativos (`product_type = 'unit'`, `is_active = true`), ordenados por `sort_order ASC`.
- [ ] AC2: Cada card de produto unitário mostra: nome e preço unitário (`R$ X,XX`). Sem imagens nesta fase. Layout legível em tela touch com dedo (altura mínima 48px por card).
- [ ] AC3: Clicar em um card de produto unitário abre um **modal de venda unitária** com:
  - Nome e preço do produto exibidos.
  - Seletor de quantidade (stepper ou input número): default 1, min 1, max 99.
  - Total calculado em tempo real: `unit_price * quantidade` (ex: "2x Picolé = R$ 6,00").
  - Seletor de método de pagamento: PIX / Cartão Crédito / Cartão Débito / Dinheiro (mesma UI do fluxo açaí).
  - Botão "Confirmar venda" e botão "Cancelar" / fechar modal.
- [ ] AC4: Ao confirmar venda unitária, registrar em `sales`:
  - `product_id` = UUID do produto selecionado.
  - `weight_grams = 0` (ou NULL se schema permitir para `unit`).
  - `price_per_gram = NULL`.
  - `weight_source = 'manual'` (ou `'NA'` — verificar constante existente no código).
  - `quantity = N` (selecionado pelo operador).
  - `amount = product.unit_price * quantity`.
  - `has_casquinha = false` (sempre, por constraint do trigger da Story 10.2).
  - `status = 'COMPLETED'`.
  - `payment_method` = método selecionado.
- [ ] AC5: Fluxo do açaí por peso **NÃO sofre nenhuma mudança de comportamento** — pesar + confirmar é zero-clique-extra. O layout novo apenas ADICIONA a área de produtos avulsos sem remover nada do fluxo atual.
- [ ] AC6: Venda unitária funciona **online e offline** — entra na fila Dexie (`pendingSales`) com payload completo e sincroniza via `sync-sales` edge function normalmente.
- [ ] AC7: Disponível para `role: 'admin'` e `role: 'staff'` (operação normal de caixa).
- [ ] AC8: Se não houver produtos unitários ativos para a location, a área secundária exibe estado vazio: "Nenhum produto avulso cadastrado. Peça ao administrador para cadastrar em Settings."
- [ ] AC9: **Catálogo em cache Dexie:** a lista de produtos unitários ativos é cacheada em uma tabela Dexie `product_catalog` com TTL simples. Cache é invalidado e refetchado quando: (a) o componente do PDV é montado (UX consistente); (b) o admin salva um produto em Settings (invalidação via query key do TanStack Query). O cache evita chamada ao Supabase a cada abertura do PDV enquanto estiver offline.
- [ ] AC10: **Edge case — produto desativado enquanto PDV está aberto:** se o admin desativar um produto enquanto o operador está com o PDV aberto, ao clicar no card desse produto exibir toast "Este produto foi desativado. Recarregando catálogo..." e disparar refetch do catálogo. O card desativado NÃO deve aparecer após o refetch.
- [ ] AC11: Render do PDV com até 30 produtos unitários ativos: < 100ms de first paint (não bloquear o carregamento do fluxo açaí).

## Dev Notes

**REGRA CRÍTICA — fluxo açaí intocado:** O AC5 é a restrição mais importante desta story. Antes de qualquer mudança em `POS.tsx`, fazer um "screenshot" mental do fluxo atual e garantir que nada do componente de açaí (input de peso, total, toggle casquinha, botão confirmar) mudou. Testes de regressão explícitos são obrigatórios.

**Localizar componentes do PDV atual:** O PDV pode estar fragmentado em múltiplos componentes. Usar:
```
grep -r "weight_grams\|price_per_gram\|confirmarVenda\|confirmSale\|scale" src/ --include="*.tsx" -l
```
para mapear todos os arquivos antes de editar.

**Cache Dexie com TTL simples:** Não usar Supabase Realtime para o catálogo — decisão arquitetural explícita do EPIC (seção Decisões em Aberto DA-3 resolvida). O TTL é implementado como `cached_at: Date.now()` salvo com o catálogo. Ao ler, verificar se `Date.now() - cached_at < 5 * 60 * 1000` (5 minutos). Se expirado, fazer fetch e atualizar o Dexie.

**Versão do Dexie:** O Dexie usa versões para migration de schema. Verificar a versão atual em `DexieDatabase.ts` (procurar por `db.version(`) e incrementar para adicionar a tabela `product_catalog`. NÃO alterar versões anteriores — apenas adicionar nova versão com o novo store.

**Seletor de pagamento reutilizável:** O fluxo de açaí já tem um seletor de método de pagamento. Extraí-lo como componente separado (se ainda não for) para reutilizar no modal de venda unitária. Isso garante UX consistente e reduz código duplicado.

**`weight_source` para produto unit:** Verificar o tipo/enum existente para `weight_source` em `src/types/index.ts`. Se existe valor `'manual'` ou similar, usar ele. Se não existe um valor semântico para "não aplicável", usar `'manual'` e documentar no PR que o campo não tem significado real para vendas unitárias mas é preenchido por consistência de schema.

**`shift_id` obrigatório:** Toda venda precisa de `shift_id` para ser associada a um turno. O fluxo de açaí atual já obtém o turno ativo. Verificar como ele obtém (`useAuthStore`, `useShiftStore`, ou query direta) e usar o mesmo mecanismo para vendas unitárias.

**Sincronização offline com `sync-sales`:** A edge function atualizada pela Story 10.2 aceita o payload com `product_id`, `quantity`, `has_casquinha`. Para vendas unitárias, o payload Dexie deve ter:
```typescript
{
  product_id: product.id,
  quantity: selectedQuantity,
  amount: product.unit_price * selectedQuantity,
  has_casquinha: false,
  weight_grams: 0,
  price_per_gram: null,
  payment_method: selectedPayment,
  shift_id: activeShift.id,
  location_id: profile.location_id,
  status: 'COMPLETED',
}
```

**Performance (AC11):** O grid de produtos não deve bloquear o carregamento da área de açaí. Usar `React.lazy` + `Suspense` para o `ProductGrid` se necessário, ou carregar o catálogo em background sem bloquear a render da área principal.
