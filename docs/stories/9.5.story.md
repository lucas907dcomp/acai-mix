## User Story

**Como** administrador da loja,  
**Quero** cancelar uma venda que ainda não foi sincronizada com o servidor (feita offline),  
**Para que** erros em vendas recentes sejam corrigidos sem aguardar reconexão — e sem que a venda incorreta chegue ao banco de dados.

---

## Acceptance Criteria

- [ ] AC1: Ao clicar em "Cancelar" em uma venda com `synced=false` em `pending_sales` (Dexie):
  - Se `syncStore.isSyncing === false`: deletar a linha de `pending_sales` via `db.pending_sales.delete(local_id)`
  - **Não** enviar UPDATE ao Supabase (a venda nunca chegou ao servidor)
  - Chamar `shiftStore.reverseTotals(amount, paymentMethod)` para atualizar totais locais
  - Chamar `syncStore.refreshPendingCount()` para atualizar contador de pendentes
  - Toast verde: `"Venda offline cancelada"`
- [ ] AC2: Se `syncStore.isSyncing === true`: **bloquear** com toast amarelo/warning: `"Aguarde a sincronização terminar"` — não executar o cancelamento
- [ ] AC3: Para venda com `synced=true` em `pending_sales` (sincronizada mas reconciliação pendente): tratar como path online — executar UPDATE Supabase normal (o ID já existe no servidor)
- [ ] AC4: `ShiftSalesTable.tsx` — vendas pendentes offline exibem ícone de relógio (`Clock` de lucide-react) ao lado do horário, indicando que ainda não foram sincronizadas
- [ ] AC5: Vendas canceladas em `ShiftSalesTable` exibem badge "Cancelada" com visual atenuado (consistente com `SalesHistoryTable` da Story 9.2)
- [ ] AC6: A lógica de detecção "offline vs online" usa `findPendingSale(saleId)` (busca por índice `id`, retorna registro completo com `local_id`). Se o registro existe E `synced === false`, é path offline; caso contrário (não está no Dexie OU `synced === true`), é path online. Ver T1 para detalhes do gotcha `local_id` vs `id`.
- [ ] AC7: Testes de integração cobrem os cenários:
  - Venda offline cancelada: `pending_sales` linha deletada, `reverseTotals` chamado, `refreshPendingCount` chamado
  - `isSyncing=true` bloqueia: nenhuma operação Dexie executada
  - Venda já sincronizada (`synced=true`): roteada para path online (UPDATE Supabase)

## Tasks

### T1 — Detecção de rota offline no handler de cancelamento (1.5h)
- [ ] Criar `src/utils/cancelSale.ts` (utilitário puro — separado do hook)
- [ ] Função: `findPendingSale(saleId: string): Promise<PendingSale | undefined>`
  ```typescript
  import { db, type PendingSale } from '@/providers/sync/DexieDatabase'

  export async function findPendingSale(saleId: string): Promise<PendingSale | undefined> {
    // 'id' é índice secundário no schema Dexie (DexieDatabase.ts linha 28/31).
    // Retornar o registro completo (com local_id) permite delete por PK depois.
    return db.pending_sales.where('id').equals(saleId).first()
  }

  export function isPendingOffline(pending: PendingSale | undefined): boolean {
    return pending !== undefined && pending.synced === false
  }
  ```
- [ ] **ATENÇÃO (gotcha Dexie):**
  - `local_id` (number) é a PK auto-incrementada — usar em `.delete(local_id)` e `.update(local_id, ...)`.
  - `id` (string UUID) é o campo da venda e é **índice secundário** no schema (ver `DexieDatabase.ts` linha 28: `'++local_id, id, ...'`) — usar em `.where('id').equals(saleId)`.
  - **Nunca:** `db.pending_sales.delete(sale.id)` — `sale.id` é string UUID, não a PK number do Dexie.
- [ ] A função retorna o registro completo (com `local_id`) para o handler poder deletar com a PK correta.

### T2 — Handler de cancelamento offline (1h)

- [ ] Antes de executar o UPDATE Supabase, executar:
  ```typescript
  const pending = await findPendingSale(sale.id)

  if (isPendingOffline(pending)) {
    // Path offline: venda nunca chegou ao Supabase
    if (useSyncStore.getState().isSyncing) {
      toast.warning('Aguarde a sincronização terminar')
      return

    useShiftStore.getState().reverseTotals(sale.amount, sale.payment_method)
    await useSyncStore.getState().refreshPendingCount()

    queryClient.invalidateQueries({ queryKey: ['shift-sales'] })
    return // NÃO continuar para o UPDATE Supabase

  // Path online (inclui pending.synced === true): fluxo da Story 9.4
  // (executa UPDATE Supabase normal)
  ```

## Dev Notes

**REGRA DA-2 — Opção A (Remover da fila Dexie):**  
Decisão arquitetural confirmada: venda offline ainda em `pending_sales` é removida do Dexie ao cancelar — nunca chega ao Supabase. Justificativa: venda offline ainda não é evento fiscal permanente. [Source: docs/architecture/fase2-tech-decisions.md#3.1]

**`local_id` vs `id` no Dexie — gotcha crítico:**  
- `id`: campo da `Sale` (UUID gerado no cliente antes do INSERT)
- `local_id`: chave primária auto-incrementada do Dexie (número inteiro)

Ao deletar de `pending_sales`, usar `db.pending_sales.where('id').equals(saleId).delete()` (busca pelo UUID da venda, não pela chave do Dexie). Alternativamente, buscar o registro completo primeiro e usar `db.pending_sales.delete(record.local_id!)`.

**Nunca fazer:** `db.pending_sales.delete(sale.id)` — o `sale.id` é UUID string, não a chave primária do Dexie (que é `local_id` number).

**Evidência no código:** Em `syncStore.ts` linha 179: `await db.pending_sales.update(sale.local_id!, { synced: true })` — o `local_id` é usado como chave para operações Dexie.

**Race condition — caso perigoso:**  
```
T0  drain() lê batch contendo venda X
T1  admin clica Cancelar → detecta como offline
T2  isSyncing ainda false (drain ainda iniciando)
T3  admin deleta venda X do Dexie
T4  drain() recebe success do Supabase para X
T5  X está no Supabase como COMPLETED, sem rastro local
```
**Mitigação:** O check `isSyncing` no T2 previne a maioria dos casos. O `drain()` seta `isSyncing=true` imediatamente ao ser chamado. Janela de race é de microssegundos. Aceitável para a fase atual. [Source: docs/architecture/fase2-tech-decisions.md#3.4]

**Consulta ao Dexie para ícone de pendente:**  
Usar `useSyncStore.getState().pendingCount` como trigger para re-consultar o Dexie (não re-consultar em loop). Quando `pendingCount` muda (nova venda offline ou drain completou), re-buscar os IDs pendentes.

**Shadcn `toast.warning`:**  
O projeto usa `sonner` para toasts (ver `saleStore.ts` linha 3: `import { toast } from 'sonner'`). Sonner suporta `toast.warning(...)` diretamente.
