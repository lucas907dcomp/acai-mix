## User Story

**Como** administrador da loja,  
**Quero** uma página de histórico com todas as vendas paginadas e filtráveis,  
**Para que** eu possa auditar operações, verificar cancelamentos e reconciliar extratos sem precisar de acesso direto ao banco.

## Dev Notes

**Padrão de rota protegida no projeto:**  
Verificar `src/App.tsx` para o padrão atual de `ProtectedRoute`. A Story 8.1 usa `/settings` como admin-only — seguir o mesmo padrão.

**Paginação com Supabase `.range()`:**  
Supabase `range(from, to)` é inclusivo em ambas as pontas. Para página 0 com 50 itens: `.range(0, 49)`. Para página 1: `.range(50, 99)`. O `count: 'exact'` retorna o total de linhas que satisfazem o filtro (sem o range), necessário para calcular `totalPages = Math.ceil(count / 50)`.

**Campo `status` em `Sale`:**  
Esta story usa o campo `status` que será adicionado pela Story 9.3. Se desenvolvida antes da 9.3 (sem a migration aplicada), a query vai retornar `status: undefined` — tratar com `sale.status ?? 'COMPLETED'` como fallback defensivo. Após 9.3, todos os registros têm `status` preenchido pelo DEFAULT da migration.

**Índice de performance:**  
A migration 9.3 cria `idx_sales_completed_location_created ON sales (location_id, created_at DESC) WHERE status = 'COMPLETED'`. Para queries que incluem canceladas (sem filtro de status), o índice completo `idx_sales_location_created` da migration 003 ainda cobre. Não criar índice extra aqui.

**AppLayout — menu admin:**  
Verificar `src/components/layout/AppLayout.tsx` para o padrão de itens de menu condicionais. Exemplo em 8.1: SettingsPage usa `role: 'admin'` guard. Usar o mesmo mecanismo.

**Date pickers — formato ISO:**  
Input `type="date"` retorna `YYYY-MM-DD`. Para construir `Date` sem problemas de timezone: `new Date(value + 'T00:00:00')` (não `new Date(value)` que interpreta como UTC).
