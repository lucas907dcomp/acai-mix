## User Story

**Como** sistema de autorização do AçaiMix,
**Quero** que o owner leia turnos/vendas/produtos/lojas de todas as suas lojas vinculadas, e que as brechas de isolamento herdadas do schema atual sejam fechadas,
**Para que** o critério de aceite original do epic — "operador não acessa dado de outra loja mesmo forjando requisição direto na API" — seja verdadeiro na prática, não só na intenção.

---

## Acceptance Criteria

- [ ] AC1: Um usuário `admin`/`staff` autenticado, mesmo fazendo requisição direta à API, não consegue ler dado de nenhuma loja fora da(s) sua(s) para `locations` e `products` — validado pelo caso **T9** (staff hipotético).
- [ ] AC2: Um usuário `owner`, via API direta, só consegue ler dados das lojas às quais está vinculado — validado pelos casos **T5** (só vê lojas vinculadas) e **T6** (loja não vinculada retorna vazio).
- [ ] AC3: A brecha adicional em `locations_update_admin` (achado do @data-engineer — admin conseguia editar qualquer loja, não só a própria) está fechada — validado pelos casos **T14** (admin edita a própria loja, zero regressão) e **T15** (admin bloqueado ao editar loja alheia).
- [ ] AC4: Owner só edita a loja **ativa no momento**, não qualquer loja vinculada — validado pelo caso **T16**.
- [ ] AC5: O teste de vazamento admin→admin em `shifts`/`sales` (**T2**) é executado e seu resultado registrado, mas **não bloqueia** esta story — é o comportamento aceito conscientemente em **DEBT-EPIC11-01** (ver epic, seção 4.2.1).
- [ ] AC6: Todas as policies de escrita existentes (`shifts_insert`, `sales_insert`, `products_insert_admin`, etc.) continuam funcionando sem alteração — owner escreve na loja ativa reaproveitando `get_my_location_id()`, sem nenhuma policy nova de escrita.

Inspecionei `qual`/`with_check` completos das 5 policies (não só os nomes):

- `locations_select_own_or_privileged`, `products_select_own_or_privileged`: `is_admin()` preservado sem escopo (zero regressão), `is_owner_of_location()` somado corretamente via `OR`. ✓
