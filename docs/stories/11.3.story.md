## User Story

**Como** dono (owner) do AçaiMix,
**Quero** um seletor de loja no header que me permita trocar qual loja estou operando,
**Para que** eu use as telas de PDV/Dashboard/Settings normalmente dentro de qualquer uma das minhas lojas, sem precisar de outro login.

---

## Acceptance Criteria

- [ ] AC1: Usuário `admin`/`staff` **não vê** o seletor — comportamento do header idêntico ao atual para eles.
- [ ] AC2: Usuário `owner` vê no header a lista de lojas vinculadas (`user_locations`) e a loja atualmente ativa.
- [ ] AC3: Ao trocar de loja no seletor, todas as telas atuais (PDV, Dashboard, Settings, Histórico) passam a operar no contexto da loja escolhida, sem exigir novo login.
- [ ] AC4: Criar uma 3ª loja fictícia e vinculá-la ao owner (via SQL direto, sem deploy) faz ela aparecer automaticamente no seletor no próximo login/refresh do profile.
- [ ] AC5: A troca de loja é **bloqueada** enquanto `useSyncStore.getState().pendingCount > 0` ou `isSyncing === true`, com mensagem explicando o motivo (DA-4).
- [ ] AC6: Ao trocar de loja com sucesso, o estado transitório de loja (`shiftStore.activeShift`, qualquer venda em andamento no PDV) é limpo antes de liberar nova interação.
- [ ] AC7: **Nenhum dos 16 hooks/componentes existentes que leem `profile.location_id` é modificado** — a troca de loja funciona porque `location_id` passa a significar "loja ativa" (decisão DA-1/DA-3), e o React Query já refaz fetch sozinho pela mudança na query key.

### Debug Log References

- **Import circular evitado:** minha primeira versão de `switchActiveLocation` colocava a guarda de sync (DA-4) e a limpeza de estado (`clearShift`/`saleStore.reset()`) dentro da própria action do `authStore`, importando `useSyncStore`, `useShiftStore`, `useSaleStore`. Como `syncStore.ts` já importa `useAuthStore` (para `_createProvisionalShift`) e `saleStore.ts` também já importa `useAuthStore` (para `confirmSale`), isso criaria 2 ciclos de import (`authStore` ↔ `syncStore`, `authStore` ↔ `saleStore`). Redesenhei: `authStore.switchActiveLocation` faz só a RPC + atualização do profile; a guarda e a limpeza ficam no componente `LocationSelector`, que já tem acesso natural a todas as stores via hooks, sem risco de ciclo.
- **Ausência de `@testing-library/react`:** o projeto não tem infraestrutura de teste de componente (todos os testes existentes são de lógica pura — stores, hooks via chamada direta, utils). Em vez de adicionar essa dependência sem aprovação, extraí a lógica de decisão pura (`satisfiesRequiredRole`, `shouldBlockLocationSwitch`) para arquivos próprios sem dependências pesadas, testáveis por chamada direta — mesmo padrão dos testes já existentes no projeto.
- **`LocationSelector.test.ts` falhava ao importar o componente inteiro** (que puxa `authStore` → `supabase.ts` → `createClient()`, que lança `"supabaseUrl is required"` porque o ambiente de teste não tem `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` definidos). Resolvido movendo `shouldBlockLocationSwitch` para `src/lib/syncGuard.ts` (zero dependências) e o teste para `src/lib/__tests__/syncGuard.test.ts` — evita puxar a cadeia de stores só para testar uma função booleana pura.
- **Achado não-relacionado a esta story:** `src/stores/__tests__/saleStore.test.ts` falha com o mesmo erro (`supabaseUrl is required`) mesmo sem nenhuma mudança minha — confirmado isolando via `git stash --include-untracked` e rodando a suíte antes de aplicar qualquer alteração desta story. É um problema pré-existente do ambiente de teste (falta `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` no processo do Vitest), não introduzido aqui. Não tentei corrigir — fora do escopo desta story, e mexer nisso sem entender por que já estava quebrado antes seria arriscado.
