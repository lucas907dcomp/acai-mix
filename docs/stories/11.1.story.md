## User Story

**Como** sistema de autorização do AçaiMix,
**Quero** um papel `owner` e uma tabela de vínculo N:N usuário↔loja,
**Para que** um único usuário possa ser associado a múltiplas lojas sem alterar em nada o comportamento dos usuários `admin`/`staff` já existentes.

---

## Acceptance Criteria

- [ ] AC1: É possível cadastrar um usuário com papel `owner` vinculado a 2 ou mais lojas (`user_locations`).
- [ ] AC2: Vincular uma nova loja a um `owner` existente é uma operação de dados (`INSERT` em `user_locations`), nunca exige alteração de código ou deploy.
- [ ] AC3: Os 2 usuários `admin` reais (simulados pelo seed sintético da Story 11.0) continuam com exatamente o mesmo comportamento, acesso e desempenho depois da migration — validado pelo smoke test (casos T1, T3, T4, T14 do arquivo `epic11_rls_smoke.sql`, aplicados no ambiente da Story 11.0).
- [ ] AC4: Nenhum identificador de loja fica hardcoded em nenhuma camada (migration, funções SQL) — validado com a 3ª loja fictícia do seed sintético (não vinculada a ninguém).
- [ ] AC5: `switch_active_location(p_location_id)` recusa (`EXCEPTION`) qualquer loja não vinculada ao owner chamador — validado pelo caso T7.
- [ ] AC6: Nenhum usuário consegue alterar o próprio `role` via `UPDATE` direto em `user_profiles` — validado pelo caso T10.
- [ ] AC7: Nenhum owner consegue alterar `location_id` fora da RPC `switch_active_location`, mesmo para uma loja legitimamente vinculada — validado pelos casos T11 e T12.
- [ ] AC8: `npx tsc -b` e `npm run lint` continuam passando após a regeneração de tipos (`supabase gen types typescript --local`).

## Data Engineer Review

### Review Date: 2026-07-15

### Reviewed By: Dara (Database Architect & Operations Engineer)

### Escopo desta revisão

Esta migration (`20260714000000_owner_multiloja.sql`) foi originalmente desenhada e revisada por mim em sessão anterior, junto com @architect. O que esta revisão cobre agora é diferente: auditar a migration já em produção-local (não redesenhar) e **verificar de forma independente** — não apenas aceitar — a evidência de validação que o @dev reportou nesta story.

### Auditoria de Segurança/Schema

- **Funções `SECURITY DEFINER`** (`is_owner`, `is_owner_of_location`, `switch_active_location`): confirmei via `pg_proc.proconfig` que as 3 têm `search_path=public` fixado — proteção padrão contra sequestro de `search_path` em funções com privilégio elevado. `fn_guard_user_profiles_update` corretamente **não** é `SECURITY DEFINER` (trigger não precisa de privilégio elevado; rodar com direitos do invocador é o comportamento certo aqui).
- **Grant de `EXECUTE` para `anon`** em `switch_active_location`: presente, mas inofensivo — a função depende de `auth.uid()` via `is_owner_of_location()`, que retorna `NULL`/`false` para sessão anônima, então a RPC só lança a exceção de "loja não vinculada". Confirmei que `is_admin()` (pré-existente) tem exatamente o mesmo padrão de grant — é o comportamento padrão do Postgres (EXECUTE herdado de `PUBLIC`), não uma regressão introduzida por esta migration.
- **Índices** (`idx_user_locations_user`, `idx_user_locations_location`): confirmados presentes via `\di`, com o discriminador de loja em primeiro no índice usado pelo padrão de leitura mais frequente, conforme a restrição do epic.
- **Idempotência**: `DROP TRIGGER IF EXISTS` + `CREATE OR REPLACE FUNCTION` em todas as funções — seguro para reaplicação. A localização dinâmica do nome da CHECK constraint via `information_schema` (em vez de assumir `user_profiles_role_check`) é a abordagem correta — evita quebra silenciosa se o nome divergir.
- Nenhum SQL dinâmico concatenado com entrada de usuário — o único `format()` usado (`ALTER TABLE ... DROP CONSTRAINT %I`) opera sobre um nome de constraint lido do próprio catálogo do Postgres, não de input externo. Sem risco de SQL injection.
