# STORY 10.3: CRUD Admin de Produtos Avulsos em Settings

**ID:** 10.3 | **Epic:** EPIC-10 — Catálogo Multi-Produto e Adicional Casquinha
**Sprint:** 7 | **Points:** 13 | **Priority:** Alta
**Created:** 2026-05-26 | **Status:** InProgress
**Executor:** @dev | **Quality Gate:** @qa | **QG Tools:** coderabbit
**Predecessor:** Story 10.2 (schema `products` com `product_type`, `unit_price`, `sort_order` deve existir)

---

## User Story

**Como** administrador do AçaiMix,
**Quero** uma tela em Settings para cadastrar, listar, editar e desativar produtos avulsos (picolé, água, refrigerante, etc.),
**Para que** eu possa montar o catálogo de produtos unitários da loja sem precisar de suporte técnico e esses produtos fiquem disponíveis no PDV para o operador.

---

## Acceptance Criteria

- [ ] AC1: Nova seção em `/settings` — aba ou card "Produtos Avulsos" (ou "Catálogo de Produtos") — visível **apenas para `role: 'admin'`**. Staff não vê essa seção.
- [ ] AC2: Lista de produtos exibe: nome, preço unitário (R$), status (badge "Ativo" / "Inativo"), `sort_order`. Ordenada por `sort_order ASC`, depois `name ASC`. Apenas produtos com `product_type = 'unit'` aparecem nesta lista.
- [ ] AC3: Botão "Novo produto" abre dialog/form com campos:
  - `name` (text, obrigatório, max 60 caracteres)
  - `unit_price` (number, obrigatório, > 0, max 2 casas decimais, mínimo R$ 0,01)
  - `sort_order` (number, opcional, default = `max(sort_order) + 10` dos produtos ativos da location)
  - `is_active` (boolean switch, default `true`)
- [ ] AC4: Formulário de criação valida client-side antes do submit:
  - Nome único por location entre produtos `unit` ativos (verificar via query antes de enviar — mostrar erro "Já existe um produto com esse nome")
  - `unit_price` deve ser número positivo com máximo 2 casas decimais
  - `name` não pode ser vazio ou só espaços
- [ ] AC5: Edição de produto via dialog (clicar em linha ou botão de edição): permite alterar `name`, `unit_price`, `sort_order`, `is_active`. Validações idênticas ao criar.
- [ ] AC6: **Não permite deletar** produto. Apenas desativar (`is_active = false`). Botão de ação: "Desativar" (ou toggle do `is_active`) — nunca DELETE. O FK `ON DELETE RESTRICT` em `sales.product_id` impede deleção acidental.
- [ ] AC7: Produto com `is_active = false` não aparece no PDV (Story 10.4). Continua listado nesta tela com badge "Inativo". Admin pode reativar via toggle/botão "Reativar".
- [ ] AC8: O produto açaí (`product_type = 'weight'`) **NÃO aparece** nesta tela. Esta tela é exclusivamente para `product_type = 'unit'`. A edição do açaí (preço/g) continua no local atual de Settings.
- [ ] AC9: `created_by` e `updated_by` gravados via `auth.uid()` nas operações de INSERT e UPDATE (colunas adicionadas pela Story 10.2 AC4).
- [ ] AC10: Feedback visual: toast de sucesso após criar/editar/desativar. Toast de erro se a operação falhar (ex: nome duplicado retornado pelo servidor, RLS negou acesso).
- [ ] AC11: Tela abre em < 500ms com até 200 produtos cadastrados. Usar TanStack Query com `staleTime` adequado.

## Tasks

- [ ] Criar `src/hooks/useProductCatalog.ts`

- [ ] Query key: `['products', 'catalog', locationId]`
- [ ] `staleTime: 30_000` (30s) — catálogo não muda frequentemente
- [ ] Retornar `{ products, isLoading, error }`

## Dev Notes

**Arquivos a criar:**
- `src/hooks/useProductCatalog.ts`
- `src/hooks/useProductMutations.ts`
- `src/components/settings/ProductList.tsx`
- `src/components/settings/ProductForm.tsx`

**Arquivos a modificar:**
- `src/pages/AdminSettings.tsx` (ou nome equivalente) — localizar com `grep -r "Settings\|settings" src/pages/ -l`

**Stack confirmada:** React 19 + TypeScript + Vite, TanStack Query, Zod, React Hook Form, Shadcn UI (já no projeto), Supabase client.

**Proteção admin-only:** A tela principal já tem proteção de rota para admin. Para a nova seção de produtos, garantir proteção dupla: (1) não renderizar o componente se `role !== 'admin'`; (2) RLS no banco impede INSERT/UPDATE por não-admins (configurado pela Story 10.2 AC13). Não usar apenas proteção de UI — o banco é a fonte da verdade.

**Invalidação de cache cross-story:** Quando admin cria/edita/desativa um produto, o catálogo do PDV (Story 10.4) deve ser atualizado. A query key `['products', 'pdv', locationId]` deve ser invalidada nas mutations. Mesmo que Story 10.4 ainda não esteja implementada, já incluir a invalidação para evitar bug futuro.

**Preço como `number` no form vs `string` no input:** Inputs HTML de tipo `number` retornam `string` no evento. Usar `valueAsNumber` do React Hook Form ou converter manualmente:
```typescript
unit_price: z.preprocess(
  (val) => parseFloat(String(val)),
  z.number().positive()
)
```

**Sort order default:** Buscar `max(sort_order)` da lista já carregada no TanStack Query (sem nova query ao banco). Se a lista estiver vazia, default = `10`.

**Produto açaí excluído da lista (AC8):** O filtro `.eq('product_type', 'unit')` na query já exclui o açaí. Não precisar de filtro adicional por nome.

**Produto inativo e PDV:** Quando admin desativa um produto (`is_active = false`), o PDV em Story 10.4 usa um cache Dexie com TTL. A invalidação da query key `['products', 'pdv', locationId]` garante que o próximo mount do PDV faça refetch. No edge case de produto desativado enquanto PDV está aberto, ver AC-F4.9 do EPIC — Story 10.4 trata isso.

**Zod + React Hook Form:** Padrão do projeto — verificar se já existe um `resolver` de Zod configurado. Modelo em outras páginas com form (verificar `src/pages/` para encontrar exemplos de form existentes no projeto).

**`created_by` / `updated_by`:** Vêm do `auth.uid()` via Supabase client. No mutation:
```typescript
const { data: { user } } = await supabase.auth.getUser();
await supabase.from('products').insert({
  ...values,
  created_by: user?.id,
  location_id: profile.location_id,
  product_type: 'unit',
})
```
