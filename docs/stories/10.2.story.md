## User Story

**Como** desenvolvedor/DBA do AçaiMix,
**Quero** aplicar uma migration segura e reversível no banco Supabase que adicione suporte a múltiplos tipos de produto e ao adicional casquinha,
**Para que** as stories 10.1, 10.3, 10.4 e 10.5 possam ser implementadas sobre um schema estável e backward-compatible com todas as vendas existentes em produção.

## Dev Notes

**Ordem de execução dos blocos SQL é crítica:** Primeiro ADD COLUMN (não bloqueante), depois UPDATE (backfill), depois validação do backfill, depois índices, depois trigger, depois RLS. Não inverter.

**Backfill multilocal:** O AçaiMix pode ter múltiplas locations. O backfill deve associar cada venda ao produto açaí da SUA location, não de outra. O subquery `WHERE location_id = sales.location_id` é essencial. Se uma location não tiver produto ativo, o subquery retorna NULL e a venda fica com `product_id = NULL` — o bloco de validação vai falhar e forçar a investigação antes do COMMIT.

**Produto açaí existente:** O projeto tem um único produto por location chamado "Açaí" (ou similar). Checar o nome exato com `SELECT name, id FROM products LIMIT 5` antes de executar o UPDATE do Bloco 3.

**Recursão em RLS — risco R4 do EPIC:** Nunca criar policy que faça `SELECT FROM profiles WHERE id = auth.uid()` diretamente dentro da policy expression. Isso causa recursão quando `profiles` também tem RLS. Encapsular em função `SECURITY DEFINER`:
```sql
CREATE OR REPLACE FUNCTION fn_get_user_location(user_id uuid)
RETURNS uuid LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT location_id FROM profiles WHERE id = user_id;
$$;
```

**Migration zero-downtime:** Todas as novas colunas têm DEFAULT ou são nullable. `ADD COLUMN` em PostgreSQL com DEFAULT constante (não uma função) é instantâneo a partir do Pg 11+ (Supabase usa Pg 15+). Sem lock de tabela prolongado.

**Edge function backward compat:** Payload antigo do PWA cacheado não tem `product_id`. A edge function DEVE resolver isso server-side:
```typescript
const productId = body.product_id ?? await resolveAcaiProductId(locationId);
const hasCasquinha = body.has_casquinha ?? false;
const quantity = body.quantity ?? 1;
```

**Função de fechamento de turno:** Localizar via `grep -r "close_shift\|fn_shift\|turno" supabase/` para encontrar o arquivo correto. Verificar se usa `sales.amount` (certo) ou recalcula via `weight_grams * price_per_gram` (errado para produtos unit).

**Arquivos-chave:**
- `supabase/migrations/` — novo arquivo de migration
- `supabase/functions/sync-sales/index.ts` — edge function
- `src/types/index.ts` — interfaces TypeScript
- `src/types/supabase.ts` — tipos gerados do schema

**Risco R2 do EPIC (alta probabilidade):** Clientes PWA cacheados em produção vão continuar mandando payload antigo por horas após o deploy da migration. A edge function DEVE estar em produção ANTES ou JUNTO com a migration. Deploy em ordem: (1) nova edge function, (2) migration SQL, (3) novo código do PWA.

## Dev Notes — Implementation

- **Coluna `active` vs `is_active`:** O texto da story menciona `is_active` em AC11, mas o schema real (migration 001) usa `active`. Mantive `active` para evitar inventar coluna; documentei no comentário do índice.
- **`fn_is_admin_of_location` vs `is_admin` existente:** O existente é global (não escopado a location). Story 10.2 explicitamente pede escopo por location, então criei o novo helper como SECURITY DEFINER (Risco R4 — sem recursão) e substituí apenas as policies INSERT/UPDATE/DELETE de products. A SELECT policy original foi preservada (staff precisa ler catálogo para PDV).
- **Cache por request na edge function:** Map por `locationId` evita N+1 quando um batch contém múltiplas vendas da mesma location sem product_id.
- **DexieDatabase v4:** Adiciona `product_catalog` sem migração de dados (rebuild from Supabase no primeiro sync online). Mantém todas as tabelas v3 intactas.
- **CHECK constraint extra em products:** Adicionei `products_unit_price_coherence` para garantir que produtos `unit` SEMPRE tenham `unit_price > 0` e produtos `weight` SEMPRE tenham `unit_price IS NULL`. Não estava explícito na story, mas é um invariante fundamental do domínio — falha cedo previne bugs caros.
- **Trigger não recursivo:** O trigger faz apenas SELECT em `products` (não UPDATE) — sem ciclo com outros triggers da tabela sales.
